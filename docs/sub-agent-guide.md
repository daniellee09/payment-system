# Sub-Agent 활용 가이드

## 개요

복잡한 작업을 병렬로 처리할 때 서브 에이전트를 활용한다.
메인 에이전트가 작업을 분할하고, 서브 에이전트에게 위임한 뒤, 결과를 조합한다.

## 활용 원칙

- 서로 독립적인 작업만 서브 에이전트로 분리한다. 의존 관계가 있으면 순서대로 처리한다.
- 서브 에이전트에게 작업을 위임할 때 CLAUDE.md의 규칙과 명세서 컨텍스트를 함께 전달한다.
- 서브 에이전트의 결과물은 메인 에이전트가 일관성을 검증한 뒤 최종 반영한다.

## 활용 시나리오

### 1. 도메인 레이어 병렬 생성

하나의 도메인(예: Product)을 구현할 때 각 레이어를 병렬로 생성한다.

```
메인: "Product 도메인을 구현해줘"

├── 서브 A: Product 엔티티 + Enum 작성
├── 서브 B: ProductRepository 인터페이스 작성
├── 서브 C: Request/Response DTO 작성
└── 메인: 결과 조합 → ProductService, ProductController 순서대로 작성
```

Service와 Controller는 다른 레이어에 의존하므로 서브 에이전트 결과 확인 후 메인이 작성한다.

### 2. 여러 도메인 테스트 병렬 작성

기존 코드가 완성된 상태에서 테스트를 일괄 작성한다.

```
메인: "전체 Service 테스트를 작성해줘"

├── 서브 A: ProductServiceTest
├── 서브 B: OrderServiceTest
└── 서브 C: PaymentServiceTest
```

각 테스트는 독립적이므로 병렬 처리에 적합하다.

### 3. 코드 리뷰 다각도 분석

작성된 코드를 여러 관점에서 동시에 검토한다.

```
메인: "payment 패키지 코드를 리뷰해줘"

├── 서브 A: 비즈니스 로직 정합성 검토 (상태 전이, 금액 검증)
├── 서브 B: 예외 처리 누락 및 에러 케이스 점검
└── 서브 C: 동시성 이슈 및 트랜잭션 경계 점검
```

### 4. 리팩토링 계획 수립

리팩토링 전에 현재 코드의 문제를 병렬로 분석한다.

```
메인: "리팩토링이 필요한 부분을 분석해줘"

├── 서브 A: 중복 코드 탐지
├── 서브 B: 책임 분리가 필요한 클래스 식별
└── 서브 C: 명세서와 현재 구현의 차이점 확인
```

### 5. 설정 파일 병렬 생성

프로젝트 초기 세팅 시 독립적인 설정 파일을 동시에 만든다.

```
메인: "프로젝트 초기 설정을 구성해줘"

├── 서브 A: build.gradle.kts (의존성 설정)
├── 서브 B: docker-compose.yml (MySQL, Redis)
├── 서브 C: application.yml (Spring 설정)
└── 서브 D: 공통 모듈 (ApiResponse, ErrorCode, GlobalExceptionHandler)
```

## 서브 에이전트에게 작업 위임 시 포함할 내용

1. **작업 범위**: 어떤 파일을 생성/수정하는지 명확히
2. **참조 문서**: 명세서의 어떤 섹션을 따르는지
3. **규칙 리마인드**: CLAUDE.md의 핵심 규칙 (Kotlin, BigDecimal, 한글 주석 등)
4. **기존 코드 컨텍스트**: 의존하는 클래스나 인터페이스가 있으면 전달
5. **완료 기준**: 어떤 상태가 되면 작업이 끝난 것인지

## 서브 에이전트를 쓰지 않는 경우

- 파일 하나만 수정하는 단순 작업
- 이전 단계의 결과에 따라 다음 단계가 달라지는 순차 작업
- 기존 코드 전체를 파악해야 하는 디버깅
